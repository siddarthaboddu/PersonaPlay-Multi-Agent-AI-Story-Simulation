# PersonaPlay Pro: System Design & Product Requirements Document

## 1. Executive Summary
**PersonaPlay Pro** is a high-fidelity, multi-agent simulation engine designed for autonomous narrative generation, improv theater, and interactive skit production. Unlike standard LLM-based chatbots, PersonaPlay Pro treats Large Language Models (LLMs) as distinct, stateful actors operating within a deterministic world environment managed by a centralized "Director" agent.

The system prioritizes **physicality**, **subtext**, and **narrative tension** to move beyond simple text generation into a full-scale digital performance suite.

---

## 2. System Architecture
The architecture follows a **Stateful Graph Orchestration** pattern, ideally implemented via frameworks like **LangGraph**, ensuring a robust control flow between the environment and the participants.

### 2.1 The "Tri-Mind" Orchestration
To prevent character drift and ensure narrative coherence, the system is segmented into three logical planes:

1.  **The Actor Plane (The Participants):** 
    * Each character is an isolated agent instance.
    * Utilizes a **Dual-Stream Prompting** architecture: Stream A (Internal Monologue/Subtext) and Stream B (Public Dialogue/Action).
    * Maintains a **Local Memory Buffer** (Last 10-15 interactions) and a **Vectorized Episodic Memory** (Long-term recall).

2.  **The Director Plane (The Orchestrator):**
    * A high-reasoning "Super-Agent" that manages turn-taking and speaker selection.
    * Monitors the **Global Tension Score** and injects "Narrative Bombs" (stochastic events) to keep the scene engaging.
    * Performs **Conflict Resolution**: If two agents attempt to use the same unique prop, the Director arbitrates the outcome.

3.  **The World Plane (The Deterministic State):**
    * A non-generative, JSON-based state machine.
    * Tracks **Inventory** (who holds what), **Spatial Zones** (who is where), and **Environmental Variables** (lighting, weather, locks).
    * Acts as the "Source of Truth" that overrides LLM hallucinations regarding physical reality.

### 2.2 Data Flow & Communication
* **Backend:** Python/FastAPI managing the state machine and agent inference.
* **Real-time Layer:** WebSockets (or SSE) to stream the "Script Feed" and "Monologue Feed" to the UI simultaneously.
* **Inference Strategy:** 
    * **Pro Models (e.g., Gemini 1.5 Pro):** Used for the Director and the active speaker's turn.
    * **Flash Models (e.g., Gemini 1.5 Flash):** Used for passive background reactions and internal monologue generation to minimize latency and token cost.

## 3. Product Requirements

### 3.1 Functional Requirements (FRs)
* **FR-1: Dynamic Ensemble Management:** Support for 3 to 10 characters in a single scene with distinct persona injection.
* **FR-2: Screenplay Format Output:** All public output must adhere to a strict screenplay schema (Scene Heading, Character Name, Parenthetical, Action, Dialogue).
* **FR-3: The Chaos Engine:** An automated trigger system that introduces external stimuli (e.g., *a phone rings*, *the police arrive*) based on a timer or tension threshold.
* **FR-4: World State Persistence:** A centralized ledger of props and character positions that persists throughout the session.
* **FR-5: Rewind & Branching:** The ability to stop the simulation, edit a character's "Hidden Agenda," and restart from a previous state to observe a different outcome.

### 3.2 Non-Functional Requirements (NFRs)
* **Latency:** Character turn-around time (including internal monologue) should not exceed 4 seconds.
* **Observability:** The system must expose the "Inner Thoughts" of agents for debugging and user engagement.
* **Context Management:** Use RAG or summarizing middleware to prevent context window overflow during long skits.

## 4. The Actor Brain: Cognitive Architecture

### 4.1 Hierarchical Memory (RAG + Episodic)
To avoid characters "forgetting" their past, each agent utilizes a two-tier memory system:
1.  **Short-term:** A sliding window of the current scene's transcript.
2.  **Long-term:** A vector database (e.g., ChromaDB) containing "Core Beliefs" and "Past Trauma" that are retrieved when relevant keywords appear in the current scene.

### 4.2 Emotion Vector State ($E$)
Each character's mood is tracked via a 4D vector:
$$E = [Tension, Affection, Energy, Suspicion]$$
* Every incoming line of dialogue from other characters updates these values.
* **Example:** A character with high `Suspicion` will interpret ambiguous dialogue as hostile.

## 5. UI/UX: The Director’s Command Center
The UI should be a **Three-Column Dashboard** designed for real-time monitoring and intervention.

### 5.1 The Theater View (Center)
* **Script Feed:** A scrolling display using a monospace "Courier" font.
* **Animated Blocking:** (Optional) A 2D grid showing icons for each character and prop moving in real-time based on `[ACTION]` tags.

### 5.2 The Backstage Panel (Right)
* **Inner Monologue Stream:** A live, translucent feed of what the character is *thinking* before they speak.
* **Agent Vitals:** Real-time graphs showing the fluctuations in the Emotion Vector ($E$).

### 5.3 The World Controller (Left)
* **Prop Ledger:** A list of items in the scene. Users can "force-give" an item to a character mid-skit.
* **Director Console:** A "God-mode" input box where the user can type global instructions like *"Make the scene much more awkward"* or *"Add a character who thinks they are a ghost."*

## 6. Implementation Roadmap

### Phase 1: MVP (The Static Duo)
* Build the LangGraph state machine for 2 agents.
* Implement basic screenplay parsing logic.
* CLI-based output showing "Thought" vs. "Action."

### Phase 2: The Ensemble & State
* Extend to 3+ characters with Director-led speaker selection.
* Implement the JSON World State tracker for props.
* Launch the first iteration of the React-based "Theater View."

### Phase 3: The Observability Layer
* Introduce the WebSocket stream for Internal Monologues.
* Implement the Emotion Vector system and UI visualizations.
* Add the "Rewind & Branch" feature.

### Phase 4: Production Polish
* Integrate TTS (Text-to-Speech) with distinct character voices.
* Add multi-scene support (INT to EXT transitions).
* Implement "Secret Agendas" for emergent storytelling.
